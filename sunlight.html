Web UI UX

合併直營診所官網

Purpose:

隨著公司業務日漸擴展，全台的直營診所數量增加，由公司統一維護的診所官網也會越來越多，除了龐大的維護成本外，在網頁內容部分也會有高度重複的情況，導致 SEO 策略分散，因此將診所集團化，便於後續管理，最終歸納出的目標：

  1. 統一品牌形象： 合併官網可以幫助公司展示統一的品牌形象。在進行合併專案前已有台北日亞美牙醫診所以及台南蒔光牙醫診所的官網，兩個網站內容相似，可能導致使用者感到困惑，不知道兩邊的資訊應以哪邊為主，尤其日後新增的官網更會加劇此狀況。透過合併，我們能以更一致的方式呈現內容，增強品牌認知度，進而建立更強大的品牌價值。
  2. 簡化使用流程及搜索： 多個網站的存在在搜尋相似的資訊時可能會有路徑迷失的情況發生。合併官網就可以將相關的內容整合在一起，簡化搜尋資訊的使用流程，讓使用者能夠更快速地找到所需的內容，降低額外的時間成本。
  3. 整合資源和提高效率： 只要新開幕一間直營診所，就需要新架設一個官方網站。但頁面架構及內容相近，這樣不僅浪費了開發成本，更讓網頁的更新和維護變得複雜。藉由合併官網，我們可以更有效地管理資源，集中精力提供更優質的內容和服務，進而提高我們的整體營運效率。

Process:

  • 比對現有診所官網架構
    • 服務項目

      在台北診所及台南診所的服務項目大致上是相同的，僅台南診所多了兒童矯正的頁面，因此在架構上關於服務項目的合併

    • 診所介紹

      在這個部分台北診所相較台南診所，多了診所環境及門診資訊 2 個頁面，其餘架構上沒有不同

    • 案例分享及最新消息

      因為兩邊的後台設計是一樣的，故無論是案例分享還是診所最新消息的資料項目也都相同

  • 整合與重構

    將相同的頁面合併，頁面內容以台北診所為準，其原因為考慮到開發過程，合併診所官網的網域為台北診所沿用而來，因此在開發上是以台北診所原始碼為基礎做修改；加上台南診所風格較童趣，有比較多的插畫配圖設計，相較台北診所風格轉換的成本高。

    在結構上將所有與診所相關的資訊都合併在該診所官方頁面，製作為獨立頁面。獨立頁面的設計也讓原官網的風格得以保留，在日後新增的診所也能有較大的空間來呈現與實體裝潢或社群貼文風格相對應的 UI 設計。

    拉出菁英醫療團隊來簡化所有醫師的資訊，因為診所端會有醫師人事的流動

風格定義

要將在專案開始前已有既定風格的台北、台南診所官網合併，並考量到日後新增的所有直營診所，主色調以黑白相間最可以達到簡約的目的，讓不同的診所頁面除了得以保留原風格外，也能透過網頁相似的元素與集團做連結，同時不會有太大的突兀感。在網頁元素的設計都由 Logo 做變形，插圖的風格也以線條為主。

1. 後台新增醫師頁面

在新的診所架構上，考量到醫師會有人事流動，因此在後台新增醫師區塊，方便行銷團隊能針對醫師人事流動的狀況做即時的網頁更新

醫師編輯項目

const DoctorsSchema = new Schema({
  name: {
    type: Schema.Types.String,
    required: true
  },
  image: {
    type: Schema.Types.String,
    default: ''
  },
  location: {
    type: [
      {
        type: Schema.Types.String,
        required: true
      }
    ],
    enum: Object.values(Location),
    required: true
  },
  title: {
    type: Schema.Types.String,
    default: ''
  },
  strength: {
    type: Schema.Types.String,
    default: ''
  },
  experience: {
    type: Schema.Types.String,
    default: ''
  },
  elite: {
    type: Schema.Types.Boolean,
    default: false
  },
  commend: {
    type: Schema.Types.String,
    default: ''
  }
})

在下方菁英團隊的部分，原本考慮了兩種方式：

  1. 直接判斷 textarea 有沒有值，有的話連動到 isAlite === true
  2. 先判斷isAlite(boolean) 的值，來決定textarea能否編輯

第 1 個方式的優勢在於可直接確保患者好評的textarea不會出現空白值，變相成為一種防呆機制，在操作部分也會減少一個步驟。不過最後我選擇第 2 個：以布林值為主，原因為較符合常態的思考邏輯，在實務討論上都是先透過基本的評估，先決定了是否將該醫師編列為首頁會出現的菁英醫師團隊後，再進行患者好評的資料搜集。

將醫師團隊轉為後台操作後提升了公司管理上的效率，過往需要瀏覽頁面才能發現有醫師資訊需要更新，現在能直接進行操作，也減少了重新部署網頁的時間成本。

const [notifyEmailForm, setNotifyEmailForm] = useState({

    location: '',
    toLocationMail: '',
    content: ''

const updateForm = e => {
    setNotifyEmailForm({

    })
  }
